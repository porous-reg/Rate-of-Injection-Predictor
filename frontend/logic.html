How the ROI webapp works

The app validates the condition against the supported grid, selects the appropriate checkpoint by injector, runs the bundled runtime, and returns the full waveform plus summary metrics.

1. Input contract

The current runtime accepts four values only:

  • injector_id: 417 or 800
  • pressure_bar: rail pressure
  • temp_c: fuel temperature
  • et_us: energizing time

Supported values come from the bundle metadata and are not extrapolated by default.

2. Validation and selection

Before prediction, the backend checks:

  • injector id is one of the supported injectors
  • pressure, temperature, and ET match the injector-specific allowed grid
  • the selected injector maps to the proper checkpoint

This keeps the UI aligned with the training bundle.

3. Runtime path

  1. Load the selected model checkpoint from the copied bundle
  2. Normalize the three condition inputs
  3. Run the model forward on CPU
  4. De-normalize the ROI waveform
  5. Return time_us and roi_mg_per_ms

The backend caches the model after first load so the next request is faster.

4. Summary metrics

The UI derives simple metrics from the predicted waveform:

  • peak ROI value
  • time of the peak
  • integrated ROI area
  • duration above peak thresholds

5. Batch mode

Batch CSV rows are parsed into the same input contract and passed through the same validation path. Invalid rows are reported individually rather than stopping the whole batch.

6. Future geometry-aware extension

v1 is condition-only. The UI and handoff are intentionally structured so a later version can add geometry-aware inputs such as injector hole count, PCD, or hole pattern without changing the app shell.
